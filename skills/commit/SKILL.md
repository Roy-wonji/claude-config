# SRP Commit Planner

변경된 파일들을 분석하고, 단일 책임 원칙(SRP)에 따라 논리적 커밋 단위로 분리한 뒤 순차적으로 커밋한다.

## 단계

### 1. 변경사항 수집

다음 명령어를 병렬로 실행하여 현재 상태를 파악한다:

- `git status` — untracked, staged, unstaged 파일 목록
- `git diff` — unstaged 변경 내용
- `git diff --cached` — staged 변경 내용
- `git log --oneline -10` — 최근 커밋 메시지 스타일 참조

변경사항이 없으면 사용자에게 알리고 종료한다.

### 2. 변경사항 분석

변경된 각 파일의 diff를 읽고 다음을 파악한다:

- 각 변경의 **목적** (새 기능, 버그 수정, 리팩터링, 테스트, 설정, 문서 등)
- 파일 간 **의존 관계** (함께 커밋해야 하는 파일 그룹)
- 하나의 파일 안에 **서로 다른 목적**의 변경이 섞여 있는지

### 3. 커밋 플랜 수립

SRP에 따라 커밋 단위를 분리한다. 각 커밋은 **하나의 목적**만 가져야 한다.

#### SRP 분리 기준

| 분리해야 하는 경우 | 예시 |
|---|---|
| 기능 추가 + 버그 수정 | 기능 A 추가 커밋, 버그 B 수정 커밋 |
| 구현 코드 + 테스트 코드 | 동일 기능이면 함께, 다른 기능이면 분리 |
| 리팩터링 + 기능 변경 | 리팩터링 먼저 커밋, 기능 변경 별도 커밋 |
| 설정/인프라 + 비즈니스 로직 | 설정 변경 커밋, 로직 변경 커밋 |
| 포맷팅/린트 수정 + 로직 변경 | 포맷팅 커밋, 로직 커밋 |

#### 함께 커밋해야 하는 경우

- 동일 기능의 구현 + 해당 테스트
- Protocol 정의 + 해당 Conformance 구현
- 모델 변경 + 해당 모델을 사용하는 코드 변경 (컴파일이 깨지지 않도록)

#### 커밋 플랜 형식

플랜을 다음 형식으로 사용자에게 제시한다:

```
## Commit Plan

### Commit 1: {타입} — {한 줄 설명}
- 파일: {파일 목록}
- 목적: {이 커밋이 하는 일}

### Commit 2: {타입} — {한 줄 설명}
- 파일: {파일 목록}
- 목적: {이 커밋이 하는 일}

...
```

타입은 다음 중 하나: `feat`, `fix`, `refactor`, `test`, `chore`, `docs`, `style`

### 4. 사용자 확인

커밋 플랜을 사용자에게 보여주고 확인을 요청한다:

- 커밋 분리가 적절한지
- 커밋 메시지가 정확한지
- 커밋 순서가 맞는지 (의존성 고려: 하위 레이어 먼저)

사용자가 수정을 요청하면 플랜을 조정한다.

### 5. 순차 커밋 실행

확인된 플랜에 따라 커밋을 순서대로 실행한다. 각 커밋마다:

1. 해당 커밋에 포함될 파일만 `git add`로 staging
2. hunk 단위 분리가 필요한 경우 (한 파일에 서로 다른 목적의 변경이 있을 때) 사용자에게 알린다
3. 커밋 메시지는 최근 커밋 스타일을 따르되, Conventional Commits 형식을 기본으로 한다
4. `git commit` 실행
5. 커밋 성공 여부 확인

### 6. 완료 보고

모든 커밋 완료 후 `git log --oneline`으로 결과를 보여준다.

## 규칙

- 커밋 메시지는 HEREDOC 방식으로 전달한다.
- `.env`, 자격 증명 파일 등 민감한 파일은 커밋하지 않는다. 발견 시 사용자에게 경고한다.
- `git add -A`나 `git add .`를 사용하지 않는다. 파일을 명시적으로 지정한다.
- 사용자가 플랜을 승인하기 전에는 커밋을 실행하지 않는다.
- push는 사용자가 명시적으로 요청한 경우에만 실행한다.
