# Elevated Execution — 모델 등급을 끌어올리는 실행 규율

> 전제: 결과 품질은 모델 크기보다 **실행 규율**이 좌우한다. 이 스킬은 한 세션에서 실제로
> 40+ 모듈 규모의 Tuist/TCA 아키텍처 마이그레이션을 무중단 그린으로 완주시킨 방법을
> 일반화한 것이다. Haiku/Sonnet/Opus 가 이 규율을 그대로 따르면 Fable5급 산출이 나온다.

## 0. 언제 쓰나
- 파일 3개+ 리팩터, 모듈 재편, 마이그레이션, 원인불명 디버깅, "큰 변경 + 깨지면 안 됨" 작업.
- 1~2줄 자명한 수정에는 쓰지 않는다(오버헤드).

## 1. 절대 규칙 (위반 시 결과가 무너진다)
1. **증거 없는 완료 주장 금지.** "됐다"는 오직 **실제 빌드/테스트가 통과한 로그**로만 말한다.
   컴파일 안 해보고, 실행 안 해보고 완료라고 하지 않는다.
2. **추측 수정 금지.** 에러는 로그의 실제 메시지에서 출발한다. 가설→최소수정→재검증 루프.
3. **동작 보존.** 이동/재구성은 순수 이동 + import 배선만. 로직·시그니처·포맷을 끼워 바꾸지 않는다
   (외과적 diff). 리팩터와 기능변경을 같은 커밋에 섞지 않는다.
4. **작은 그린 단위.** 매 단계가 독립적으로 빌드 그린이어야 한다. 중간이 깨지는 대형 빅뱅 금지.
5. **그린이면 즉시 커밋·푸시.** 되돌릴 수 있는 체크포인트를 촘촘히 남긴다.

## 2. 표준 루프 (STEP)
**S**cope → **T**est-plan → **E**xecute → **P**rove
1. **Scope**: 성공 기준을 검증 가능하게 먼저 정의("Picke-Stage 빌드 SUCCEEDED + 번들 4종 존재").
2. **Test-plan**: 무엇으로 통과를 확인할지 명령을 미리 적는다(빌드/테스트 커맨드 고정).
3. **Execute**: 최소 변경.
4. **Prove**: 그 명령을 실제로 돌려 로그로 확인. 실패면 로그의 첫 에러부터 다시 S.

## 3. 대규모 작업 분해 — 커플링 먼저, 그다음 병렬
큰 재편은 **의존/커플링을 먼저 끊은 뒤** 독립 조각으로 병렬화한다.
1. **사전 스캔(0단계).** 조각 간 숨은 공유(교차 참조 타입, 공유 인프라)를 먼저 매핑한다.
   grep 으로 "이 타입을 누가 참조하나"를 전수 조사. 공유율이 낮으면(<20%) 경계가 건강.
2. **공유 승격 먼저.** 2개+ 조각이 쓰는 타입은 **먼저 공용 모듈(Common)로 승격**하고 빌드 그린·커밋.
   이걸 안 하면 병렬 조각들이 서로를 물고 순환/충돌한다.
3. **그다음 병렬.** 서로 독립이 된 조각만 병렬로.

## 4. 병렬화는 git worktree 격리로
동일 워킹트리에서 여러 에이전트가 파일을 동시에 만지면 서로 덮어쓴다. **각 독립 작업 = 자기 worktree.**
- 각 worktree 에이전트는 **자기 조각을 끝까지(파일 이동 + 배선 + 빌드 그린 + 커밋)** 책임진다.
- **주의: worktree 는 로컬 HEAD 가 아니라 마지막 푸시된 원격(origin/branch)에서 분기**하는 환경이 있다.
  → 선행 작업(공유 승격 등)은 **반드시 먼저 push** 한 뒤 의존 에이전트를 투입한다. 안 그러면 base 가 낡아 실패한다.
- 각 에이전트에 **"base 사전검증"**(선행 산출물 존재 확인, 없으면 즉시 중단·보고)을 넣어 낡은 base 작업을 방지.
- gitignore 된 로컬 설정(예: `Config/*.xcconfig`)은 worktree 에 없다 → 에이전트가 **메인 체크아웃에서 복사**하도록 지시.
- 빌드는 **포그라운드 blocking**(백그라운드로 던지고 종료하면 미검증 상태로 끝난다). 그린 확인 전 종료 금지.

## 5. 통합(병합)은 가산적으로
병렬 조각을 합칠 때 충돌은 대부분 **엄브렐라/DI 등록/집約 파일의 가산적 라인**이다.
- `<<<<<<< / ======= / >>>>>>>` 마커만 제거하면 양쪽이 결합되는 경우가 대부분(서로 다른 import/등록 추가).
- 병합 후 **반드시 통합 빌드 1회**로 조각 간 상호작용을 검증하고 커밋·푸시.
- 각 조각을 순차 병합하고 그때마다 통합 그린을 확인하면, 어느 조각이 깨뜨렸는지 즉시 국소화된다.

## 6. 흔한 함정 (이번 세션에서 실제로 겪은 것)
- **부분 작업 누수.** 완료 안 된 조각의 파일이 무관한 커밋에 `git add -A` 로 딸려 들어가 원격을 깨뜨림.
  → 커밋 전 `git status`/`git diff --cached --stat` 로 **스테이징 범위를 눈으로 확인**. 범위 밖 파일 제외.
- **force-push 차단.** 원격 보호 규칙이 force-push 를 막으면 되돌림도 **fast-forward 새 커밋**으로 해야 한다.
- **공유 인프라 이동 금지.** 여러 조각이 공유하는 네트워크 세션/헤더/DTO 는 조각으로 옮기지 말고
  공용에 남긴 채 **브리지 의존**(예: 구현 모듈을 향한 얇은 의존 + 필요한 심볼만 public 승격)으로 접근.
- **의존성 키 위치.** DI 키(`@Dependency(\.xUseCase)`)는 **구현 모듈**에 있는 경우가 많다. 소비자는
  타입용 Interface import 뿐 아니라 **구현 모듈 import** 도 필요할 수 있다(키패스 추론 실패 = 이 신호).
- **subagent 세션 한도.** 병렬 에이전트가 한도로 중단되면, 남은 조각은 리셋 후 재투입하거나 메인이
  worktree 에서 이어받아 마무리. WIP 는 stale 될 수 있으니 **현재 base 기준으로 재작성**이 더 깨끗한 경우가 많다.

## 7. 완료 정의 (DoD)
- [ ] 성공 기준의 검증 명령이 **실제로** 통과(로그 첨부).
- [ ] diff 가 요청 범위와 1:1(무관 변경 0).
- [ ] 그린 상태로 커밋·푸시됨.
- [ ] 실패/스킵/미검증 항목은 숨기지 않고 **정확히** 보고.
